## Repository Structure

This is a Go microservices repository using Go workspaces. The main structure follows:

- **Go Workspace**: Uses `go.work` to manage multiple modules (currently Go 1.25.1)
- **symbol-service/**: A Kratos-based microservice for symbol management
  - Built using the [Kratos framework](https://go-kratos.dev/) for microservices
  - Follows Clean Architecture with distinct layers: api, internal (biz, data, service, server), cmd
  - Uses Protocol Buffers for API definitions and code generation
  - Supports both HTTP and gRPC transports
- **docker-compose.yml**: Docker Compose configuration for local development (MySQL + symbol-service)

## Build and Development Commands

All commands should be run from the `symbol-service/` directory:

### Essential Commands
```bash
# Initialize development environment (install required tools)
make init

# Generate API code from proto files
make api

# Generate internal proto code
make config

# Generate all Go code (wire dependency injection, etc.)
make generate

# Build the service binary
make build

# Generate all files (api + config + generate)
make all

# Run the service (after building)
./bin/symbol-service -conf ./configs

# Show all available make targets
make help
```

### Development Workflow
1. Modify `.proto` files in `api/` or `internal/` directories
2. Run `make all` to regenerate all code (api + config + wire)
3. Run `make build` to compile
4. Run database migrations: `cd symbol-service && go run cmd/migrate/migrate.go -conf ./configs/config.yaml`
5. Use `./bin/symbol-service -conf ./configs` to run locally

### Docker Workflow
```bash
# Using Docker directly (from symbol-service/ directory)
docker build -t symbol-service .
docker run --rm -p 8000:8000 -p 9000:9000 -v ./configs:/data/conf symbol-service

# Using Docker Compose (from repository root)
# Set environment variables in .env file: MYSQL_ROOT_PASSWORD, MYSQL_DATABASE
docker-compose up -d
docker-compose logs -f symbol-service
docker-compose down
```

## Architecture Overview

### Kratos Framework Structure
- **api/symbol/v1/**: Protocol Buffer definitions and generated API code
  - `symbols.proto`: Main service API definition with HTTP/gRPC mappings
  - `error_reason.proto`: Custom error definitions
  - Generated files: `*_grpc.pb.go` (gRPC), `*_http.pb.go` (HTTP), `*.pb.go` (messages)
  - Uses `validate` proto extensions for field validation
- **internal/**: Business logic following Clean Architecture (layers depend downward only)
  - `biz/`: Business logic layer (use cases, domain models, repository interfaces)
    - `symbol.go`: Domain logic for symbol operations
    - `symbol_validator.go`: Custom validation rules using go-playground/validator
  - `data/`: Data access layer (implements repository interfaces, database/cache operations)
    - `symbol.go`: Repository implementation with GORM
    - `data.go`: Data layer initialization, database connection, migration support
  - `service/`: Service layer (implements proto service interfaces, calls biz layer)
    - `symbol.go`: gRPC/HTTP service implementation
    - `symbol_transformers.go`: Data transformation between layers
  - `server/`: Transport layer (HTTP/gRPC server initialization and middleware)
    - `http.go`: HTTP server configuration
    - `grpc.go`: gRPC server configuration
  - `conf/`: Configuration proto definitions and generated structs
- **cmd/**: Command-line applications
  - `symbol-service/`: Main application entry point
    - `wire.go`: Wire dependency injection configuration (defines provider sets)
    - `wire_gen.go`: Auto-generated by `make generate` (do not edit manually)
    - `main.go`: Application bootstrap (loads config, runs Wire, starts servers)
  - `migrate/`: Database migration tool
    - `migrate.go`: Runs GORM AutoMigrate for database schema updates
- **configs/**: YAML configuration files
  - `config.yaml`: Local development configuration
  - `config.docker.yaml`: Docker container configuration
- **third_party/**: Third-party proto dependencies (google/api, validate, etc.)
- **bin/**: Compiled binaries (generated by `make build`)

### Service Configuration
- HTTP server: Port 8000
- gRPC server: Port 9000
- Database: MySQL (configured for local development)
- Cache: Redis
- Configuration via YAML files in `configs/`

### Code Generation and Dependency Flow
The project heavily relies on code generation. Always run in this order:
1. **Protocol Buffers**:
   - `make api` generates from `api/**/*.proto` → `*_grpc.pb.go`, `*_http.pb.go`, `*.pb.go`, `openapi.yaml`
   - `make config` generates from `internal/conf/conf.proto` → `conf.pb.go`
2. **Wire**: `make generate` runs `go generate ./...` which:
   - Scans for `//go:generate` directives
   - Executes `wire` in `cmd/symbol-service/` to generate `wire_gen.go`
   - Creates the dependency injection wiring based on `ProviderSet` declarations
3. **Build**: `make build` compiles the final binary

**Wire Provider Sets** (defined in each layer's main file):
- `service.ProviderSet` → provides service implementations
- `biz.ProviderSet` → provides use cases and validators
- `data.ProviderSet` → provides repositories and data clients
- `server.ProviderSet` → provides HTTP/gRPC servers

**Validation**: Uses `go-playground/validator/v10` for runtime validation and proto `validate` extensions for request validation

## Key Tools and Dependencies
- **Kratos v2.8.0**: Microservice framework (provides HTTP/gRPC servers, middleware, config loading)
- **Wire 0.6.0**: Compile-time dependency injection (generates `wire_gen.go`)
- **Protocol Buffers**: API definition and code generation
  - `protoc-gen-go`: Generates Go structs from proto files
  - `protoc-gen-go-grpc`: Generates gRPC client/server code
  - `protoc-gen-go-http`: Generates Kratos HTTP handlers
  - `protoc-gen-openapi`: Generates OpenAPI/Swagger documentation
  - `protoc-gen-validate`: Generates validation code for proto messages
- **gRPC 1.65.0**: RPC communication
- **GORM 1.31.0**: ORM for database operations (with MySQL driver 1.6.0)
- **MySQL 8.0**: Primary database (via gorm.io/driver/mysql)
- **go-playground/validator/v10.27.0**: Runtime validation
- **Go 1.24.0**: Required Go version (workspace uses Go 1.25.1)
- **automaxprocs 1.5.1**: Automatically sets GOMAXPROCS to match Linux container CPU quota

## Database Migrations
The project uses GORM AutoMigrate for schema management:
- **Migration Command**: `go run cmd/migrate/migrate.go -conf ./configs/config.yaml`
- **How It Works**: Runs `db.AutoMigrate()` on all entity structs in `internal/data/`
- **When to Run**: After adding/modifying entity structs or database schema changes
- **Migration Logic**: Located in `internal/data/data.go` (Data.Migrate() method)

## Important Development Notes
- Never manually edit generated files (`*.pb.go`, `wire_gen.go`, `openapi.yaml`)
- Always run `make all` after modifying proto files before building
- Run `go mod tidy` after adding new dependencies (automatically done by `make generate`)
- The service uses both HTTP RESTful API and gRPC (dual transport)
- Proto validation rules are enforced automatically via middleware
- Clean Architecture: dependencies flow inward (service → biz → data)
- Use the `cmd/migrate` tool for database schema updates, not manual SQL scripts
- never change go version
